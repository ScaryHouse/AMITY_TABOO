AMITY TABOO — UI-FIRST PROJECT PLAN (PATH B) — NOTES UPDATED

PHASE 1 — UI + GAMEPLAY (DO FIRST)

1. DEFINE CORE GAME BEHAVIOR
A) What happens when BEGIN PLAY is pressed.
B) How a 100-card deck is created (shuffle, optional seed).
C) How cards advance (tap, button, or timer).
D) End-of-deck behavior at Card 100.

2. UI STATE MODEL
State A: Ready to play.
State B: Playing — card N of 100.
State C: Finished.
For each state, specify:
- Visible text.
- Enabled buttons.
- Screen-reader focus target.

3. IMPLEMENT GAMEPLAY (STILL HOSTED BY APPS SCRIPT)
A) Generate deck on BEGIN PLAY.
B) Render current card in the white field.
C) Add Next Card logic.
D) Test with keyboard, touch, and a screen reader.

4. MILESTONE LOCK
Create a ZIP snapshot and versioned deployment when gameplay feels correct.


PHASE 2 — ARCHITECTURE SPLIT (DO LATER)

5. CREATE SEPARATE UI PROJECT
Local folder example:
Documents/Amity-Taboo-UI/
- phone.html
- app.js
- assets (if any)

6. REFACTOR APPS SCRIPT TO API-ONLY
Expose JSON endpoints such as:
- ?action=newDeck
- ?action=nextCard&id=...

7. CONNECT EXTERNAL UI TO APPS SCRIPT
Replace google.script.run with fetch() calls to the Apps Script URL.

8. HOST THE UI
Choose one:
- Cloudflare Pages
- Netlify
- GitHub Pages


PHASE 3 — HOME-SCREEN / PWA CONSIDERATIONS (IDEAS TO REMEMBER)

9. ADD TO HOME SCREEN (A2HS) INSTRUCTIONS
A) On first load, show a simple, high-contrast instruction panel above or within the card.
B) Android instruction example:
   "Tap the three dots (⋮) in Chrome → Add to Home screen."
C) iPhone instruction example:
   "Tap Share → Add to Home Screen."
D) Hide these instructions once the user taps BEGIN PLAY.

10. BIG LOGO AS VISUAL CUE
A) Consider placing a very large AMITY TABOO logo or mark at the top of the first screen.
B) This can double as a clear visual target and an attractive home-screen icon preview.

11. OPTIONAL: MAKE IT A PWA
A) Add manifest.json with name, short_name, and high-resolution icons.
B) Add a service worker for basic offline support.
C) Expect better consistency on Android than iOS, but both will still allow “Add to Home Screen.”


RATIONALE

- Finish design and behavior first.
- Avoid solving gameplay and hosting/CORS simultaneously.
- Split architecture only after the game works well.
