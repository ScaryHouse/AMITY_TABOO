WHAT THIS PROJECT IS

Amity Taboo is an accessible Taboo-style prompt web app.

It is intended to deliver a simple, single-screen card experience with no teams, no timers, and no scorekeeping.

The current implementation is a Google Apps Script web app with HTML UI.


PROJECT PURPOSE

This project exists to finish and maintain the Amity Taboo game so it can be used reliably by real players.

The priority is a working, accessible gameplay experience first.

Architecture split, external hosting, and PWA work are later-path improvements, not the first finish line.


CURRENT PRIORITY

Finish the hosted Apps Script version so gameplay feels correct before tomorrow afternoon.

That means the immediate focus is the playable phone UI and card-flow behavior, not larger platform refactoring.


PRODUCT BEHAVIOR

The app should present a clear, accessible, single-screen prompt experience.

Core characteristics:

1. Browser-based web app.

2. Single-screen card UI.

3. No teams.

4. No timers.

5. No scorekeeping.

6. Card content should fit the viewport without requiring scrolling when possible.


GAMEPLAY MODEL

The gameplay path described in the recovered planning docs is:

1. User starts in a ready state.

2. User presses BEGIN PLAY.

3. A 100-card deck is generated.

4. The current card is rendered in the white field.

5. User advances through cards one at a time.

6. End-of-deck behavior must be explicit at Card 100.

Recovered planning notes identify these required state models:

A. Ready to play
B. Playing — card N of 100
C. Finished

For each state, the UI should define:
1. visible text
2. enabled buttons
3. screen-reader focus target
